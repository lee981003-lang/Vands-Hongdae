# AGENTS.md — 협업 규칙

> 이 파일은 **Claude**와 **Codex**가 이 프로젝트에서 협업하는 방식을 정의한다.
> 작업 시작 전 반드시 이 파일을 읽고, 아래 규칙을 따른다.

---

## 1. 역할 분담 (가장 중요)

| 에이전트 | 역할 |
|---|---|
| **Claude** | **계획 담당.** 계획 수립, 추가·수정, 설계 결정 정리, 리스크 검토. 직접 구현 코드는 작성하지 않는다. |
| **Codex** | **구현 담당.** Claude가 수립·확정한 계획을 **기반으로** 실제 프로그램을 구현한다. 계획 자체를 새로 바꾸지 않는다. |

- 계획을 바꿔야 할 일이 생기면 → **Claude가 계획을 수정**하고 Notion에 반영한 뒤, Codex가 그 변경분을 읽고 구현한다.
- Codex는 구현 중 발견한 문제·제안을 **계획 변경 요청**으로 Claude에게 전달한다(직접 계획 문서를 갈아엎지 않는다).

---

## 2. 계획의 위치 (Single Source of Truth)

- **모든 계획은 Notion에 기록한다.**
- 메인 계획 페이지: **[실시간 시술 현황 대시보드 — 구현 계획](https://www.notion.so/382fe9425b5581c58634cf5bc794e576)**
  - Page ID: `382fe942-5b55-81c5-8634-cf5bc794e576`
- 계획의 최신 확정 사항·변경 이력은 해당 페이지의 **「결정 로그」** 섹션에서 확인한다.

---

## 3. 읽기 규칙 (필수)

> **전체를 읽지 않는다. 항상 필요한 부분만 좁게 읽고 진행한다.**

- 계획이나 수정사항을 읽을 때 **페이지 전체를 통째로 읽지 말 것.**
- 지금 작업에 직접 관련된 **해당 섹션/항목만** 좁게 읽는다.
  - 예: 데이터 모델 작업 → "1. 데이터 모델 & 백엔드"만.
  - 예: 타이머 로직 → "3. 핵심 로직"만.
  - 예: 최근 변경 확인 → "결정 로그"만.
- 이유: 컨텍스트 절약 + 작업 정확도 유지 + 불필요한 재해석 방지.
- Notion `fetch`/검색 시에도 관련 키워드·섹션 단위로 좁혀서 가져온다.

---

## 4. 작업 흐름

```
Claude(계획 수립/수정) → Notion 기록 → Codex(해당 부분만 읽고 구현)
                ↑                                  │
                └──── 계획 변경 요청 ◀──────────────┘
```

1. Claude가 계획을 수립/수정하고 Notion에 반영한다.
2. Codex는 구현할 부분에 해당하는 **Notion 섹션만 좁게** 읽는다.
3. 구현 중 계획과 어긋나거나 결정이 필요하면 Claude에게 계획 변경을 요청한다.
4. Claude가 계획을 갱신 → Codex가 변경분만 읽고 반영한다.
